## 一、通用检查项

> 适用于所有需求，无论涉及哪个模块

### 1.1 需求完整性

- [ ] 需求描述是否清晰完整？
- [ ] AC（验收标准）是否明确可验证？
- [ ] 是否有遗漏的功能点？
- [ ] 边界条件是否明确定义？
- [ ] 异常/错误场景是否考虑？

### 1.2 兼容性影响

- [ ] 是否影响已有功能的行为？
- [ ] 是否需要数据库迁移？
- [ ] 是否影响其他模块？（参考 dependencies.md）
- [ ] 是否影响已发布的 API 接口？

### 1.3 非功能性要求

- [ ] 性能要求是否明确？
- [ ] 是否需要支持多语言？
- [ ] 安全性/权限是否考虑？

## 二、模块特有检查项

### 2.1 购物车（shopping-cart）

> 当需求涉及 购物车、加购、结算、SKU 时，检查以下项目

- [ ] 是否考虑了并发加购的幂等性？
- [ ] 价格计算是否使用整数分（cent）而非浮点？
- [ ] 是否包含库存二次校验（加购时+结算时）？
- [ ] 登录后购物车合并策略是否明确？

### 2.2 订单（order）

> 当需求涉及 订单、下单、退款 时，检查以下项目

- [ ] 订单状态机流转是否完整定义？
- [ ] 是否考虑了支付超时自动取消？
- [ ] 退款流程是否需要审批？
- [ ] 订单数据归档策略是否明确？

### 2.3 支付（payment）

> 当需求涉及 支付、付款、退款 时，检查以下项目

- [ ] 是否对接了多支付渠道？
- [ ] 支付回调的重试和幂等是否处理？
- [ ] 对账机制是否明确？
- [ ] PCI DSS 合规要求是否满足？

### 2.4 商品（product）

> 当需求涉及 商品、SKU、分类、搜索 时，检查以下项目

- [ ] SKU 属性组合是否有上限？
- [ ] 搜索索引更新策略是否明确？
- [ ] 图片 CDN 策略是否考虑？
- [ ] 是否考虑了商品上下架的缓存失效？

---

## 三、跨模块检查项

### 3.1 高风险依赖链路

- [ ] 涉及 购物车→订单 链路时，结算数据传递的完整性是否验证？
- [ ] 涉及 订单→支付 链路时，支付状态回调是否正确更新订单？
- [ ] 涉及 商品→购物车 链路时，商品价格变更是否实时同步到购物车？
