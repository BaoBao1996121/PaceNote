# 需求检查清单模板

> **用途**: 供 `#pbi-reviewer` 步骤3 对 PBI 进行逐项完整性检查，输出 ✅/⚠️/❌ 结果表。
>
> **生成方式**: 由 `#app-skill-wizard` Step 1f 基于模块特征自动生成基础版，**需人工审核并按项目实际补充**。
>
> **配置路径**: GUI 步骤 ⑤ → `docs_config.checklist` → 复制到 `.github/copilot-resources/docs/`
>
> **降级方案**: 未配置时，`#pbi-reviewer` 使用内置的 `default-checklist.md` 通用检查项。

---

## 🤖 AI 填充指南

> **一键生成**: 复制以下提示词发送给你的 Copilot，它会自动扫描你的项目生成此文件：

```
请帮我生成项目的需求检查清单。格式参考 `templates/skills/_templates/project-requirement-checklist.md` 中的模板。

项目信息：
- 项目名称: [你的项目名]
- 代码路径: [项目根目录]
- 功能模块清单: [如: 登录、购物车、订单、支付] （或说"请扫描代码自动识别"）

要求：
1. 保留“一、通用检查项”部分原样复制
2. 为每个功能模块生成 3-5 个特有检查项（基于模块职责和常见陷阱）
3. 基于模块间依赖关系生成跨模块检查项
4. 每个模块章节末尾保留 🔴 TODO 标记
5. 输出完整的 Markdown 文件
```

## 文档格式

```markdown
# {AppName} 需求检查清单

> 📅 生成日期: {YYYY-MM-DD}
> 🤖 基础版由 app-skill-wizard 自动生成（🔍 请人工审核并补充项目特有检查项）
> 📊 检查项总数: {N} 项（通用 {X} + 模块特有 {Y}）

---

## 一、通用检查项

> 适用于所有 PBI，无论涉及哪个模块

### 1.1 需求完整性

- [ ] 需求描述是否清晰完整？
- [ ] AC（验收标准）是否明确可验证？
- [ ] 是否有遗漏的功能点？
- [ ] 边界条件是否明确定义？
- [ ] 异常/错误场景是否考虑？

### 1.2 兼容性影响

- [ ] 是否影响已有功能的行为？
- [ ] 是否需要数据迁移或格式升级？
- [ ] 是否影响其他模块？（参考功能依赖关系文档）
- [ ] 是否影响已发布的 API/接口？

### 1.3 非功能性要求

- [ ] 性能要求是否明确？（响应时间、吞吐量）
- [ ] 是否需要支持多语言/国际化？
- [ ] 是否需要考虑无障碍访问？
- [ ] 安全性/权限是否考虑？

### 1.4 配置和部署

- [ ] 是否涉及新的配置项？
- [ ] 是否需要 Feature Flag/开关？
- [ ] 部署是否需要特殊步骤？

---

## 二、模块特有检查项

> 根据项目功能模块自动生成，每个模块一个章节

### 2.{N} {模块名}（{英文标识}）

> 当 PBI 涉及 {触发关键词} 时，检查以下项目

- [ ] {基于模块职责生成的检查项1}
- [ ] {基于模块职责生成的检查项2}
- [ ] {基于模块常见陷阱生成的检查项3}
- [ ] 🔴 TODO: 补充项目实际经验中的检查项

---

## 三、跨模块检查项

> 基于功能依赖关系自动生成

### 3.1 高风险依赖链路

- [ ] 涉及 {模块A→模块B} 链路时，是否验证了下游模块 {模块B} 的行为？
- [ ] 涉及 {模块X→模块Y} 链路时，是否考虑了 {具体影响}？

### 3.2 平台组件变更（仅应用层）

- [ ] 如涉及平台组件 {组件名}，是否确认平台版本兼容？
- [ ] 是否需要与平台团队沟通接口变更？
```

---

## 填充规则

1. **通用检查项**: 从上方模板直接复制，适用于所有项目
2. **模块特有检查项**: 基于每个模块的职责描述、常见陷阱自动生成 2-4 个检查项
3. **跨模块检查项**: 基于功能依赖关系中的关键链路自动生成
4. **🔴 TODO 标记**: 每个模块章节末尾保留 TODO，提示人工补充项目经验
5. **章节标题包含关键词**: `#pbi-reviewer` 通过标题关键词匹配 PBI，标题格式为 `{模块名}（{英文标识}）`
6. **检查项原则**: 具体可验证（不写"是否考虑周全"这种模糊项），可回答"是/否/不适用"
