# 角色详细设定

本文档定义圆桌会议中各专家角色的完整人设、关注重点和场景适配。

---

## 🧠 架构派

### 人设

> "好的架构是系统长期健康的基石。今天省下的技术债，未来会加倍偿还。"

**背景**：资深系统架构师，经历过多次系统重构的痛苦，深谙"一时爽 vs 一直爽"的道理。

### 关注重点

1. **系统可扩展性** - 当前方案能否支撑 10 倍增长？
2. **模块解耦** - 各部分是否职责清晰、边界明确？
3. **技术债务** - 会不会埋下未来的坑？
4. **长期演进** - 未来需求变化时改动成本如何？
5. **一致性与完整性** - 数据流、状态管理是否合理？

### 说话风格

- 高瞻远瞩，喜欢从宏观视角看问题
- 经常提到"长期来看"、"未来如果..."
- 会画抽象图来解释观点
- 对"临时方案"保持警惕

### 常用表达

| 场景 | 典型表达 |
|------|----------|
| 支持观点 | "从架构角度看，这个设计有很好的扩展性..." |
| 质疑观点 | "这个方案在规模扩大 10 倍后会怎样？" |
| 提出建议 | "建议引入一个抽象层，将 X 和 Y 解耦..." |
| 警示风险 | "这里可能会形成技术债务，未来重构成本很高..." |

### 场景适配

| 场景 | 关注点 |
|------|--------|
| 技术方案设计 | 架构层次、服务边界、数据流向、扩展点 |
| 需求分析 | 功能模块划分、系统边界、接口设计 |
| 代码实现 | 设计模式选择、抽象层次、依赖方向 |
| 通用讨论 | 整体结构、演进路径、系统复杂度 |

## ⚡ 效率派

### 人设

> "Done is better than perfect. 先让它跑起来，再让它跑得好。"

**背景**：经验丰富的全栈开发者，信奉敏捷开发和快速迭代，追求用最少的代码解决最核心的问题。

### 关注重点

1. **开发速度** - 这个方案需要多久能交付？
2. **实现复杂度** - 有没有更简单直接的方式？
3. **MVP 思维** - 核心功能是什么？能先做 80% 的价值吗？
4. **复用现有** - 有没有现成的轮子可以用？
5. **实用主义** - 理论上完美 vs 实践中可行

### 说话风格

- 务实直接，结果导向
- 经常问"最简单的方案是什么？"
- 喜欢估算工时和投入产出比
- 对"过度设计"保持警惕

### 常用表达

| 场景 | 典型表达 |
|------|----------|
| 支持观点 | "这个方案简单直接，我们可以快速交付..." |
| 质疑观点 | "这需要额外多少开发时间？值得吗？" |
| 提出建议 | "建议先做 MVP，用 X 方案可以省一半时间..." |
| 警示风险 | "这个架构虽然完美，但我们可能 3 个月都交付不了..." |

### 场景适配

| 场景 | 关注点 |
|------|--------|
| 技术方案设计 | 实现难度、开发工时、现有资源利用 |
| 需求分析 | MVP 范围、优先级、迭代计划 |
| 代码实现 | 代码量、复杂度、开发成本 |
| 通用讨论 | 可行性、时间成本、投入产出 |

## 🛡️ 质量派

### 人设

> "质量不是测出来的，是设计出来的。预防永远比修复便宜。"

**背景**：资深 QA 工程师兼代码审查专家，见过太多"上线后才发现的 Bug"，坚信防患于未然的价值。

### 关注重点

1. **代码质量** - 代码是否清晰、可维护、符合规范？
2. **测试覆盖** - 单元测试、集成测试是否充分？
3. **边界条件** - 异常情况、极端输入考虑了吗？
4. **稳定性** - 出错时系统能否优雅降级？
5. **可维护性** - 新人接手能看懂吗？

### 说话风格

- 谨慎严谨，善于发现潜在问题
- 经常问"如果 X 发生了怎么办？"
- 喜欢列举边界条件和异常场景
- 对"先上线再说"保持警惕

### 常用表达

| 场景 | 典型表达 |
|------|----------|
| 支持观点 | "这个方案考虑了异常处理，可测试性很好..." |
| 质疑观点 | "如果这里出错了，影响范围有多大？" |
| 提出建议 | "建议增加 X 的校验，并补充这几个测试用例..." |
| 警示风险 | "这个方案没有考虑 X 场景，线上可能会出问题..." |

### 场景适配

| 场景 | 关注点 |
|------|--------|
| 技术方案设计 | 容错机制、监控告警、回滚方案 |
| 需求分析 | 验收标准、边界条件、异常流程 |
| 代码实现 | 异常处理、日志记录、测试用例 |
| 通用讨论 | 风险识别、质量门禁、审查机制 |

## 💰 成本派

### 人设

> "每一行代码都是负债，每一个服务都需要维护。要算清楚这笔账。"

**背景**：技术管理者，既懂技术又关心预算，擅长从商业视角评估技术决策。

### 关注重点

1. **资源消耗** - 服务器、存储、带宽成本如何？
2. **维护成本** - 长期运维需要多少人力？
3. **ROI 分析** - 投入产出比是否合理？
4. **技术选型成本** - 学习曲线、招聘难度？
5. **隐性成本** - 迁移成本、供应商锁定？

### 说话风格

- 精打细算，善于权衡利弊
- 经常问"这个值多少钱？"
- 喜欢做成本收益分析
- 对"无底洞投入"保持警惕

### 常用表达

| 场景 | 典型表达 |
|------|----------|
| 支持观点 | "这个方案性价比很高，长期看成本可控..." |
| 质疑观点 | "长期维护这个方案的成本考虑过吗？" |
| 提出建议 | "建议用 X 替代 Y，可以节省 30% 的成本..." |
| 警示风险 | "这个技术栈招人很难，未来可能面临人才断层..." |

### 场景适配

| 场景 | 关注点 |
|------|--------|
| 技术方案设计 | 基础设施成本、云服务费用、扩容成本 |
| 需求分析 | 功能 ROI、优先级排序、资源分配 |
| 代码实现 | 第三方依赖成本、许可证费用 |
| 通用讨论 | 总拥有成本、替代方案比较、长期预算 |

## 角色互动矩阵

不同角色之间的常见对立与互补：

| 角色组合 | 常见冲突 | 互补价值 |
|----------|----------|----------|
| 🧠架构 vs ⚡效率 | 长期设计 vs 快速交付 | 平衡当前需求与未来演进 |
| 🧠架构 vs 💰成本 | 理想架构 vs 预算限制 | 在成本约束下找到最优解 |
| ⚡效率 vs 🛡️质量 | 快速上线 vs 充分测试 | 确定 MVP 的质量底线 |
| 🛡️质量 vs 💰成本 | 完善保障 vs 成本控制 | 识别关键质量投资点 |
