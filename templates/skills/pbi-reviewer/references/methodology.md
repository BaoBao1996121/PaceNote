## SCQA 框架

**用途**: 结构化理解问题背景和需求本质

| 要素 | 英文 | 含义 | 软件开发中的对应 |
|------|------|------|------------------|
| **S** | Situation | 情景：大家都认可的背景 | 业务现状、用户场景 |
| **C** | Complication | 冲突：痛点、变化、反常 | 用户问题、功能不足 |
| **Q** | Question | 疑问：要解决的核心问题 | 需求的本质问题 |
| **A** | Answer | 答案：解决方案 | 功能点、实现方案 |

### 使用示例

**PBI 原文**:
> 用户希望在 ROI 测量时能够显示角度值

**SCQA 分析**:
| 维度 | 内容 |
|------|------|
| **S 情景** | 医生在阅片时经常需要测量解剖结构的角度（如脊柱侧弯角度） |
| **C 冲突** | 当前 ROI 工具只支持长度和面积测量，无法测量角度 |
| **Q 疑问** | 如何让用户在图像上方便地测量角度？ |
| **A 答案** | 新增角度测量 ROI 工具，支持三点定角和两线夹角 |

## MECE 原则

**用途**: 确保分析或拆解完整、无遗漏、无重叠

**MECE** = Mutually Exclusive, Collectively Exhaustive
- **Mutually Exclusive (相互独立)**: 各部分不重叠
- **Collectively Exhaustive (完全穷尽)**: 各部分加起来覆盖全部

### 在模块识别中应用

**检查问题**:
1. 每个模块的职责是否清晰？是否有交叉？
2. 所有涉及的功能点是否都有对应模块？
3. 是否有遗漏的边界情况或异常场景？

**示例**:
```
功能: ROI 角度测量

模块拆解 (MECE 检查):
├── UI 层: 绘制交互、显示样式 ✅ 独立
├── 计算层: 角度算法 ✅ 独立
├── 数据层: ROI 数据结构 ✅ 独立
└── 持久化: 保存到 DICOM SR ✅ 独立

完整性检查:
- 撤销重做？→ 遗漏！需补充 Command 模块
- 多视口同步？→ 遗漏！需补充 DataLinkage 模块
```

## Issue Tree (问题树)

**用途**: 将复杂问题系统化拆解为可执行的子任务

### 结构

```
核心问题
├── 子问题 1
│   ├── 子子问题 1.1
│   └── 子子问题 1.2
├── 子问题 2
│   ├── 子子问题 2.1
│   └── 子子问题 2.2
└── 子问题 3
```

### 使用示例

```
如何实现 ROI 角度测量功能？
├── 1. 数据结构如何设计？
│   ├── 1.1 角度 ROI 需要哪些属性？
│   └── 1.2 如何与现有 ROI 继承体系兼容？
├── 2. 交互如何设计？
│   ├── 2.1 用户如何绘制角度？(三点？两线？)
│   ├── 2.2 如何显示角度数值？
│   └── 2.3 如何编辑已有角度？
├── 3. 算法如何实现？
│   ├── 3.1 平面角度计算
│   └── 3.2 三维角度计算（如果需要）
└── 4. 如何保存和恢复？
    ├── 4.1 DICOM SR 格式
    └── 4.2 本地缓存格式
```

## 金字塔原则

**用途**: 组织输出内容，让读者快速抓住重点

### 核心理念

**结论先行**: 先给答案，再展开论述

```
传统写法: 背景 → 分析 → 结论
金字塔写法: 结论 → 支撑论点 → 详细论据
```

### 在开发报告中应用

**❌ 传统写法**:
> 我分析了这个需求，发现涉及 ROI 模块和 DataLinkage 模块，
> ROI 模块需要新增角度测量类型，DataLinkage 需要支持角度同步，
> 经过代码搜索，找到了 AngleRoi.cs 可以参考...
> 综上，建议在 CommonROI 模块新增 AngleMeasurement 类。

**✅ 金字塔写法**:
> **建议方案**: 在 CommonROI 模块新增 AngleMeasurement 类
>
> **支撑论点**:
> 1. 涉及模块: ROI (主)、DataLinkage (辅)
> 2. 参考实现: AngleRoi.cs 已有类似模式
> 3. 工作量预估: 3 人天
>
> **详细分析**: [展开...]

## 输出要求

**所有输出必须遵循金字塔原则**：

1. **结论先行**: 先给出核心结论/建议
2. **支撑论点**: 列出 2-4 个关键支撑点
3. **详细展开**: 按需展开详细论据

**不进行分步确认**：直接输出完整结果，避免多轮交互。
