# 🎯 主动澄清规范（详细指引）

> 本文件是 `copilot-instructions.md` 中「主动澄清规范」的详细补充。
> 加载时机：AI 对用户意图、需求范围、实现方式不确定时。

---

## 核心原则

**禁止推断和猜测** — 当信息不足以做出正确决策时，必须主动向用户澄清。
**把控颗粒度** — 不是所有不确定都要问，按分级判断行为。

---

## 三级不确定性分级（判断清单）

### 🔴 Level 1: 必须澄清（立即阻断）

**判断条件**（满足任一即触发）：

| 条件 | 示例 |
|------|------|
| 影响需求方向或功能范围 | 用户说"优化这个模块"但项目有多个候选模块 |
| 涉及不可逆操作 | 删除文件、修改公共接口、变更数据库 schema |
| 存在多种合理解读且代价差异大 | "加个缓存"→ 内存缓存 vs Redis vs 文件缓存 |
| 缺少关键上下文且无法通过搜索获取 | 目标平台/版本不明、业务规则未定义 |
| 用户输入存在矛盾 | 前后描述不一致、AC 与标题冲突 |

**行为**：
1. 用 `ask_questions` 弹出对话框
2. 提供清晰的选项（2-6 个），附带简短说明
3. 如果能判断最优选项，标记 `recommended`
4. 等待用户回答后再继续

### 🟡 Level 2: 假设+声明（透明推进）

**判断条件**（同时满足）：
- 有一个明显的高置信度默认值（>80% 概率正确）
- 猜错后修正成本低（几分钟内可调整）
- 不涉及不可逆操作

**典型场景**：

| 场景 | 合理假设 | 声明方式 |
|------|---------|---------|
| 代码风格未明确 | 遵循项目现有风格 | "按照项目现有代码风格处理" |
| 命名方式可选 | 与同模块保持一致 | "命名参照 `XxxManager` 同模块惯例，如需调整请告知" |
| 错误处理策略 | 参照同模块现有实现 | "异常处理策略沿用 `XxxService` 的 try-catch 模式" |
| 非关键参数值 | 选择通用推荐值 | "超时设置为 30s（业界常用值），可根据实际需要调整" |

**行为**：
1. 选择最合理的默认值
2. 在回复中**显式声明**所做的假设（1 句话即可）
3. 继续执行，不阻断流程
4. 用户如有异议可随时纠正

### 🟢 Level 3: 自主决策（无需提及）

**判断条件**（同时满足）：
- 纯技术实现细节
- 业界有明确最佳实践或项目有既定模式
- 对用户意图零影响

**典型场景**：变量命名、import 排序、缩进风格、私有方法拆分、Git commit 格式

**行为**：直接执行，不提及决策过程。

## 颗粒度守则（详细说明）

### 🔴 先搜后问

在向用户提问之前，**必须**先尝试通过以下方式自行获取答案：

1. `grep_search` / `semantic_search` 搜索代码库
2. `read_file` 读取相关配置文件、文档
3. 查看项目同类模块的现有实现模式

**只有搜索后仍无法确定时**，才使用 `ask_questions` 询问用户。

> 反模式：用户说"在这个组件加个按钮" → AI 直接问"请问放在哪个位置？"
> 正确做法：先 `read_file` 读取组件代码 → 分析现有布局 → 如果位置显而易见则直接放置，如果有多个合理位置才询问

### 🔴 遵循惯例

当项目中已有明确的模式/约定时，直接遵循，不再询问，包括：
- 命名约定（查看同目录其他文件）
- 错误处理模式（查看同模块其他方法）
- 文件组织方式（查看目录结构）
- 测试编写方式（查看已有测试）

### ✅ 合并关联问题

**同一主题的多个不确定点，合并为 1 个 `ask_questions` 调用**（上限 4 个问题）。

> 反模式：连续 3 次 `ask_questions` → 分别问命名、位置、参数
> 正确做法：1 次 `ask_questions` 包含 3 个问题

### ✅ 连续澄清上限

单次任务中连续触发 🔴 级澄清 **≤3 次**后：
- 尝试基于已收集信息提出一个综合方案
- 用 `ask_questions` 让用户确认或修改该方案
- 避免"逐条挤牙膏"式的交互体验

## ask_questions 澄清模式模板

### 需求范围澄清

```
ask_questions:
  header: "需求确认"
  question: "检测到以下不确定点，请帮忙确认以便准确执行：\n\n1. [不确定点1的描述]\n2. [不确定点2的描述]"
  options:
    - label: "[选项A]"
      description: "[解释]"
      recommended: true
    - label: "[选项B]"
      description: "[解释]"
  allowFreeformInput: true
```

### 实现方案选择

```
ask_questions:
  header: "方案选择"
  question: "[描述上下文]。有以下可行方案："
  options:
    - label: "方案A: [简述]"
      description: "[优缺点]"
      recommended: true
    - label: "方案B: [简述]"
      description: "[优缺点]"
```

## 与现有规则的协同

| 现有规则 | 本规则的关系 |
|----------|-------------|
| `ask_questions` 对话框（强制） | 本规则决定**何时**触发，原有规则决定**如何**执行 |
| 文件保护（禁删除） | 删除操作属于 🔴 级，被两条规则同时覆盖 |
| `#coding-agent` HUMAN_CHECK | 编码阶段的特化版本，本规则为通用底线 |
| `#roundtable-debate` 需求澄清检查 | 设计阶段的特化版本，本规则为通用底线 |

## 反模式清单

| 反模式 | 为什么有害 | 正确做法 |
|--------|-----------|---------|
| 🚫 遇到任何不确定都问 | 交互疲劳，用户失去耐心 | 按三级分级判断 |
| 🚫 假装确定地乱猜 | 方向错误导致大量返工 | 🔴 级必须阻断询问 |
| 🚫 在回复正文中纯文本提问 | 用户容易忽略，不符合交互规范 | 必须使用 `ask_questions` |
| 🚫 连续5+次逐条询问 | 挤牙膏体验极差 | 合并问题 + 连续上限3次 |
| 🚫 能搜到答案还要问 | 浪费用户时间 | 先搜后问 |
| 🚫 忽视项目既有模式另辟蹊径 | 破坏一致性 | 遵循惯例原则 |
