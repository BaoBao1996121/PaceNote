# 🔄 研发工作流阶段过渡指引

> 三阶段工作流（需求返讲 → 设计评审 → 编码）之间的过渡注意事项。
> 每个 Skill 完成时自动输出下一步建议，以下是核心检查点。

## 阶段过渡矩阵

| 从 → 到 | 过渡检查 | 重要性 |
|----------|---------|--------|
| PM → 需求返讲 | 确认 PBI 内容是否为最终版本 | ⭐⭐ |
| 需求返讲 → 设计评审 | **🔴 强制交互确认：待确认问题是否已全部与 PM 沟通并得到答复？** | ⭐⭐⭐ |
| 设计评审 → 编码 | 设计方案是否经过团队评审？关键接口是否明确？ | ⭐⭐⭐ |
| 编码 → 完成 | HUMAN_CHECK 标记是否全部确认？建议编译验证 | ⭐⭐ |

## 🔴 一次性请求拦截规则

当用户在**同一请求中**同时提到「需求返讲」和「设计评审/圆桌会议」时，**必须**使用 `ask_questions` 弹框引导分步执行：
- 建议先完成需求返讲，与 PM 确认所有待澄清问题后，再启动设计评审
- 用户可选择跳过确认直接连续执行，但默认推荐分步
- 具体拦截逻辑见 `#pbi-reviewer` Skill 的「前置检查」章节

## 核心原则

1. **未澄清的问题 = 设计偏差的根源**：需求中的待确认问题若不与 PM 确认就进入设计，方案可能偏离
2. **未评审的设计 = 编码返工的风险**：设计未经团队确认就开始编码，可能导致方向性错误
3. **过渡引导为建议性的**：AI 会在阶段完成时弹出提示，用户可一键跳过
